INTRODUCTION Undergraduate students in computing and engineering disciplines routinely produce academic artifacts—IEEE conference papers, project proposals, progress reports, and presentation slides—each governed by distinct structural and formatting conventions. Mastery of LaTeX, bibliography management, and venue-specific templates constitutes a steep learning curve that diverts effort from technical content. Existing tools address fragments of this problem: Overleaf provides collaborative LaTeX editing but requires manual template management; reference managers handle citations but not document structure; general-purpose LLM chatbots generate prose but lack guaranteed formatting compliance. SmartDoc AI addresses this gap by unifying three capabilities in a single system: (1) a conversational interface that elicits required content through 3–5 targeted questions, (2) an AI-powered template system that enforces 100% formatting compliance for supported academic formats, and (3) a project dashboard that organizes multi-document workflows. The system is designed for undergraduate students (Semesters 3–8) and operates at zero operational constraints: free-tier hosting, sub-10-minute end-to-end generation, and fewer than five clicks to first document. This paper contributes a system description of SmartDoc AI, focusing on the architecture and design decisions that enable reliable formatting compliance through LLM-guided template instantiation. We present the conversation flow, template engine, edge-function orchestration, and deployment model. As a design and architecture paper, we do not report user-study data; evaluation metrics are specified as future work. II. RELATED WORK Academic writing support tools span several categories. Collaborative LaTeX platforms (Overleaf, ShareLaTeX) provide real-time editing and template galleries but retain form-based project creation and manual formatting responsibility [1]. Reference management systems (Zotero, Mendeley) automate citation insertion but do not govern document skeleton or section-level structure [2]. General-purpose LLM assistants (ChatGPT, Claude, Mistral Le Chat) generate coherent text from prompts but exhibit hallucinated section headings, inconsistent bibliography styles, and no guarantee of venue-specific compliance [3]. Specialized academic writing aids (Paperpile, Typeset) offer template-aware authoring but rely on structured forms rather than conversational elicitation [4]. Conversational interfaces for structured document generation have been explored in requirements engineering [5] and legal document drafting [6], where slot-filling dialogues populate predefined schemas. SmartDoc AI adapts this pattern to academic templates, coupling a lightweight question-answering loop with a template engine that compiles LaTeX source and PDF output server-side. The use of edge-deployed functions for LLM orchestration and LaTeX compilation follows emerging serverless AI application patterns [7], [8]. III. SYSTEM DESIGN A. Design Goals The system targets four primary objectives derived from student workflows: 1) Generation Time: Average time to produce a complete IEEE conference paper < 10 minutes. 2) Formatting Compliance: 100% adherence to supported template specifications (IEEE conference, proposal, report, Beamer slides). 3) Interaction Friction: < 5 clicks from landing page to first generated document. 4) User Satisfaction: Target > 4.5/5 (specified as a design goal; validation deferred). Secondary goals include persistent project history, mobile-responsive access, and zero operational cost during the academic development semester. B. User Personas and Flows Three personas guide feature prioritization: - Rahul (CS Junior): Needs rapid IEEE paper generation without LaTeX expertise. - Priya (Group Leader): Manages multiple documents (paper, slides, proposal) across a team project. - Alex (Research Student): Switches between output formats for the same technical content. The core flow (Fig. 1) begins with a natural-language request (“I need an IEEE paper for my AI project”). The AI asks 3–5 clarifying questions (topic, contribution, methodology, results, target conference). Upon completion, the system generates LaTeX source, compiles PDF, and offers one-click derivation of sibling artifacts (proposal, slides, code documentation). C. High-Level Architecture The architecture (Fig. 2) comprises three layers: Client Layer: React 18 + Tailwind CSS on Vercel. Provides responsive chat interface, project dashboard, document preview, and download controls. Supabase Layer: Edge Functions (Deno/TypeScript) for serverless logic—chat handling, document generation, export, project CRUD, template loading. PostgreSQL database for users, projects, documents, and templates. Supabase Storage for PDF/LaTeX artifacts. Supabase Auth reserved for Phase 2. External Services: Mistral AI API (free tier, 10K tokens) for content generation and conversation. All API keys secured in Edge Function secrets. Communication occurs via HTTPS/REST. The frontend invokes Edge Functions; Edge Functions call Mistral and manage database/storage operations. D. Template System and Formatting Compliance The central novelty is an AI-powered template system that guarantees formatting compliance through three mechanisms: 1) Template Registry: Each supported document type (IEEE Paper, Proposal, Progress Report, Beamer Slides, Custom) stores a structured prompt in the database, encoding required sections, LaTeX class options, bibliography style, and mandatory metadata fields. 2) Structured Elicitation: The chat-handler Edge Function drives a deterministic question sequence per template (5 questions for IEEE, 4 for Proposal, 3 for Report, 4 for Slides). User answers are validated for presence before generation. 3) Constrained Generation: The document-generator Edge Function constructs a system prompt that combines the template specification with the collected answers, instructing Mistral to produce only the LaTeX source conforming to the template. The output is compiled via a LaTeX engine (server-side) to PDF; both artifacts are stored and versioned. This design isolates formatting knowledge in the template prompts and compilation pipeline, preventing the LLM from deviating from required structure. IV. IMPLEMENTATION A. Frontend The React application (Vite 5, React Router 6) implements two primary views: - Chat View: Persistent message history, quick-reply buttons for conference selection and confirmation actions, progress indicator during generation, inline preview/download actions. - Project Dashboard: Card-based project list with progress badges (0/25/50/75/100%), document status chips (Draft/In Progress/Complete), quick actions (Edit, Download PDF/LaTeX/Word, Share, Delete), project-level ZIP export, search/filter. UI components use Lucide React icons; date formatting via date-fns. The chat input remains fixed at viewport bottom on mobile and desktop. B. Backend: Edge Functions Six Edge Functions implement the serverless backend (Table I). | Function | Path | Responsibility | |----------|------|----------------| | chat-handler | /functions/v1/chat | Conversation state, question sequencing, Mistral Q&A calls | | document-generator | /functions/v1/generate | Template prompt assembly, Mistral generation call, LaTeX compilation, artifact storage | | document-export | /functions/v1/export | Format conversion (PDF, LaTeX, Word) from stored source | | project-manager | /functions/v1/projects | CRUD for projects and document metadata | | template-loader | /functions/v1/templates | Template registry retrieval | | auth-middleware | — | JWT verification (Phase 2) | The chat-handler maintains conversation context in the documents table. When the question sequence completes, it invokes document-generator asynchronously, returning a generation ID for polling. The document-generator loads the template prompt, injects user answers, calls Mistral with a system prompt enforcing LaTeX-only output, compiles via a LaTeX runtime (TeX Live in the Edge Function container), writes .tex and .pdf to Supabase Storage, and updates the document record with URLs and status. C. Database Schema Key tables (PostgreSQL): - profiles: Extended user attributes (name, avatar, institution). - projects: id, user_id, name, description, created_at, due_date. - documents: id, project_id, template_type, status (draft/in_progress/complete), progress (0–100), latex_url, pdf_url, conversation_json, created_at, updated_at. - templates: id, key (ieee_paper, proposal, report, slides, custom), system_prompt, question_sequence (ordered array), estimated_time_sec. Row-level security policies restrict access to owning user. D. Deployment and Cost Model All services operate on free tiers (Table II). | Service | Platform | Cost | Deploy Trigger | |---------|----------|------|----------------| | Frontend | Vercel | Free | Git push | | Edge Functions | Supabase | Free | `supabase functions deploy` | | Database | Supabase PostgreSQL | Free | Auto-provisioned | | Storage | Supabase Storage | Free | Auto-provisioned | | AI API | Mistral | Free tier (10K tokens) | API key in secrets | This satisfies the zero-cost development constraint. V. FEATURES A. Feature 1: Smart Q&A Document Generator The conversational generator reduces cognitive load by replacing form fields with dialogue. The AI asks exactly the questions required by the selected template—no more, no less. Quick-reply buttons accelerate closed-choice steps (e.g., conference selection: ICC, ICRA, INFOCOM). A progress indicator (“Generating: 75%”) provides transparency during compilation. Inline actions avoid modal dialogs. B. Feature 2: One-Click Document Templates The template gallery presents five cards (IEEE Paper, Proposal, Report, Slides, Custom). Selecting a template seeds the chat with the appropriate question sequence. The template engine ensures that each document type produces output matching its specification: IEEEtran class with conference option for papers; article class with custom sectioning for proposals; Beamer class for slides. Generation times are bounded (15–30 seconds) by the constrained prompt and server-side compilation. C. Feature 3: Project Dashboard Projects group related documents (paper, proposal, slides, code docs). The dashboard shows aggregate progress, per-document status, due dates, and bulk actions (ZIP export). This supports Priya’s multi-artifact workflow and Alex’s format-switching need. VI. DISCUSSION A. Achieving Formatting Compliance The 100% compliance target for supported templates is achieved by moving formatting authority out of the LLM’s free generation and into the template prompt + compilation pipeline. The LLM receives a system prompt that explicitly forbids markdown, explanatory text, or structural deviation. The LaTeX compilation step acts as a verifier: compilation failures trigger a single retry with an error-injection prompt. This design trades LLM flexibility for deterministic output—a necessary constraint for academic venues. B. Conversation Design Limiting questions to 3–5 per template balances completeness with friction. The question sequences are derived from template metadata requirements (title, authors, abstract, keywords, sections, bibliography). Optional fields (acknowledgments, appendices) are omitted from the core flow but remain accessible via the Custom template or post-generation editing. C. Scalability and Latency Edge Functions execute near the user, minimizing round-trip latency for chat interactions. Mistral API latency dominates generation time (typically 2–5 seconds for the constrained prompt). LaTeX compilation adds 5–15 seconds. The asynchronous pattern with polling keeps the chat responsive. D. Limitations and Future Work - No authentication in Phase 1; anonymous sessions limit persistence across devices. - Evaluation metrics (generation time, satisfaction, compliance) are specified but not yet measured. - Template coverage is limited to five types; extensibility requires prompt engineering per venue. - Word export relies on post-hoc conversion (pandoc) and may not preserve advanced LaTeX constructs. - Collaborative editing (Priya’s team use case) is not supported in the current architecture. Planned Phase 2 work includes Supabase Auth integration, real-time collaborative cursors, template marketplace, and a formal user study with the target undergraduate population. VII. CONCLUSION SmartDoc AI demonstrates that an AI-powered template system, combining conversational elicitation with constrained LLM generation and server-side LaTeX compilation, can deliver formatting-compliant academic documents at low friction and zero operational cost. The architecture—React/Vercel frontend, Supabase Edge Functions/PostgreSQL/Storage backend, Mistral AI for content—exemplifies a modern serverless AI application pattern suitable for student-tool development. The system’s primary contribution is the template engine design that guarantees compliance while preserving the natural-language interaction students prefer. Future work will validate the design goals through controlled user studies and expand the template registry to cover additional conferences and journals.
